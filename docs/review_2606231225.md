# ARTICLE-DRAFT.md レビュー (2026-06-23)

**対象**: 初心者向けハンズオン記事（Amplifyを少し触った程度の読者）
**レビュー観点**: 再現性・迷子になりにくさ・説明の十分さ

## 全体評価

- 再現性を非常に重視した構成（git tag + 作業ブランチ + verifyスクリプト + snapshotコピー推奨）は**優秀**で、初心者ハンズオンとして適切な方向性。
- 「AmplifyのTodoをin-placeでBlocks化」というコンセプト自体が分かりやすく、対比表やアーキテクチャ図も良い。
- ただし、**「今どの状態にいるのか」「なぜこの操作をするのか」** がやや不足しており、Amplify経験が浅い読者が迷子になりやすいポイントが散見される。
- 特に **git操作** と **Chapter 1開始時点の状態** が最大のリスク。

---

## 優先度高：強く改善を推奨する箇所

### 1. git tag / 作業ブランチのワークフロー（最大のつまずきポイント）

**問題**:
- 読者想定が「フロントエンド寄りの開発者」であるにもかかわらず、`git switch -c work/chapter1 phase-1-blocks-scaffold` というやや上級者向けの操作を各章の冒頭で要求している。
- detached HEADの警告はあるが、「なぜタグから作業ブランチを切るのか」「編集後にどうやって確認するのか」が初心者には伝わりにくい。
- 実際に `git switch` で失敗したり、どのブランチにいるか分からなくなったりするケースが予想される。
- `git diff --exit-code chapter-X` で完成形と比較できるのは素晴らしいが、その前に迷子になる可能性が高い。

**提案**:
- 各章の冒頭に「この章の作業フロー（30秒で把握）」というボックスを追加。
  ```markdown
  ### この章の作業の流れ（必ずこの順で）
  1. `git switch -c work/chapter1 phase-1-blocks-scaffold` で作業ブランチ作成
  2. スナップショットをコピー（または手で編集）
  3. `npm run dev` で確認
  4. `git diff --exit-code chapter-1-minimal-crud` で完成形と一致するか確認
  ```
- 「迷子になったら（回復手順）」を troubleshooting の冒頭に専用セクションとして追加。
  - `git branch --show-current` で現在位置確認
  - `git switch -C work/chapterX <tag名>` でその章の開始状態に強制リセットする方法
- 可能であれば、冒頭の「ハンズオンの進め方」セクションに簡単なgitフロー図（テキストベースでも可）を入れる。

### 2. Chapter 1開始時点のUI状態（Authenticatorの有無）が不明確

**問題**:
- Phase 0終了時点では「Authenticatorのログイン画面が表示される」のが成功条件。
- しかし第1章の開始は `phase-1-blocks-scaffold` からで、「Authenticator はまだ付けない」と明記されている。
- **この時点でUIにログイン画面が出るのか出ないのか** が読者には分からない。
- ch1のコードは `window.prompt` や入力欄を使い、認証なしでTodo操作をする設計のはずだが、起点のApp.tsxがAuthenticatorを残したままなのか、外しているのかが明記されていない。
- 結果として「ch1でログインを求められて詰まる」トラブルが起きやすい。

**提案**:
- 「1-1. 起点に戻る」の直後に以下の説明を追加：
  > **この時点のUIの状態**: Authenticator（ログイン画面）は**表示されません**。  
  > 単純な「+ new」ボタンまたは入力欄があるTodoアプリの画面になっています。  
  > （Phase 0のAmplify版とは異なり、phase-1-blocks-scaffold時点で認証UIは外されています）
- 可能なら、phase-1-blocks-scaffold時点のスクリーンショットを1枚追加（または「ch1開始時点ではログイン不要」と明記）。

### 3. `content` → `title` へのフィールド名変更の理由が説明不足

**問題**:
- ch1では `content` だったフィールドが、ch2で突然 `title` に変わる。
- ch3でさらに `priority` / `version` が追加される。
- 読者は「なぜ変えるのか？」「既存のTodoデータはどうなるのか？」と疑問に思う。
- 特にDynamoDBのpartitionKeyを `todoId` → `userId` に変えるch2の変更と合わせて、**スキーマ変更の影響**が気になりやすい。

**提案**:
- ch2-2の冒頭に1〜2行の説明を追加：
  > この章で `content` を `title` に変更するのは、**第3章で優先度ソートや楽観的ロックを追加するため**です。  
  > このハンズオンではSandboxのDynamoDBデータは検証後に削除する前提なので、フィールド変更による既存データへの影響は無視して構いません。
- 可能なら、ch1→ch2のスキーマ変更点を表でまとめる（Before/After）。

### 4. コード内の技術的要素（Zod / Array.fromAsync / requireAuth）の説明が薄い

**問題**:
- 初心者読者にとって以下の点が自明でない：
  - `z.object({ ... })` で何をしているのか（Zodを知らない可能性）
  - `await Array.fromAsync(todos.scan())` ではなく普通の `await` ではダメなのか
  - ch2で突然出てくる `auth.requireAuth(context)` はどこで定義され、何をしているのか
- 「丸ごとスナップショットをコピーしてください」で再現性は保てるが、「なぜこう書くのか」の理解が shallow になりやすい。

**提案**:
- 各コードブロックの直後に「補足」として短い説明を入れる（1〜3行で十分）。
  - Zod: 「`zod` というライブラリで、Todoデータの形（型とバリデーション）を定義しています。これにより不正なデータが保存されるのを防ぎます。」
  - Array.fromAsync: 「`todos.scan()` は『非同期でデータを少しずつ返す仕組み（非同期イテレータ）』を返します。全部を配列にするために `Array.fromAsync` を使っています。このチュートリアル程度のデータ量では問題になりません。」
  - requireAuth: 「`CognitoVerifier` がIDトークンを検証し、成功した場合に `user.sub`（CognitoのユーザーID）を取り出します。これを `userId` として保存することで、ログインユーザーごとのTodo分離を実現しています。」

### 5. SandboxをPhase 0で起動し続けることの意味・影響の強調が不足

**問題**:
- 「本記事の推奨ルート」では Phase 0 の Sandbox を第3章まで起動し続ける。
- その結果、**第1章でも実際は本物のDynamoDBにデータが保存される** ことが、詳細ボックスに書いてあるものの、目立っていない。
- 読者が「ch1はローカルmockで試せる」と思い込んで混乱する可能性。

**提案**:
- Phase 0の直後と、第1章の冒頭で以下の注意を強調：
  > **重要**: この記事では Phase 0 で起動した Sandbox をそのまま使い続けます。  
  > そのため **第1章のTodo作成・一覧も、実際にはあなたのAWSアカウントのDynamoDBに保存**されます（リポジトリ内の `.bb-data/` ローカルファイルではありません）。  
  > これにより第2章への移行がスムーズになります。

### 6. `amplify_outputs.json` が何を格納しているのかの説明

**問題**:
- 「File written: amplify_outputs.json」が出たら次に進む、とあるが、このファイルが**Cognitoの接続情報やBlocks APIのエンドポイントURL**を保持していることが明記されていない。
- 後で「client.jsがこのファイルを読む」と出てくるが、最初に何者なのかを軽く説明した方が良い。

**提案**:
- Phase 0-2の成功条件の近くに追加：
  > このファイルには **Cognito User Poolの情報** と **Blocksが提供するAPIのURL** が自動で書き込まれます。  
  > これを元にフロントエンド（`aws-blocks/client.js`）がCognitoログインやAPI呼び出しを行います。  
  > （gitにはcommitしないでください）

### 8. ch3 Realtime (`subscribeTodos`) の補足

- 「Amplifyの `observeQuery` に相当」とあるのは良いが、**なぜsubscribeが失敗する場合があるのか**（Realtime Blockの設定、Sandboxの制限など）と、**失敗してもlist/createは動く**ことをもう少し明示的に。
- 「subscribeTodos がエラーになってもUIの基本操作は `load()` でフォールバックする」と既に書いてあるのは親切なので、そのまま活かす。

### 9. トラブルシュートに追加してほしい項目

- 「gitで迷子になったときの回復手順」（前述）
- 「変更を反映したのにUIが変わらない」→ 「Sandboxの `✔ Deployment completed` を待ってからブラウザを再読み込みしたか確認」

### 10. その他細かい指摘

- **Nixの理由**: 「再現性のためNode.jsバージョンを固定できるから推奨」と一言追加すると良い（すでに書いてあるが短く）。
