## 🔴 致命的: 第2章タグと第3章タグが完全に同一（章の境界が壊れている）

これが本ハンズオン最大の再現性問題です。

### 事実（検証済み）

```
$ git diff --stat chapter-2-cognito-auth chapter-3-advanced
（出力なし＝完全に同一）

$ diff 03-chapter2-cognito-auth/snapshots/index.ts 04-chapter3-advanced/snapshots/index.ts
IDENTICAL index.ts
```

- `chapter-2-cognito-auth` タグの `aws-blocks/index.ts` には、**すでに第3章相当の機能が全部入っています**:
  - `Realtime` / `subscribeTodos`
  - `toggleTodo` / `deleteTodo`
  - `listTodos(sortBy?)` と Secondary Index（`byPriority`, `byTitle`）
  - `priority` / `completed` / `version`（楽観的ロック）
- `chapter-1 → chapter-2` の差分パッチ（`03-chapter2-cognito-auth/diffs/from-chapter-1.patch`）にも、これらすべてが含まれています。
- 第3章用の差分パッチ（`from-chapter-2.patch`）は**存在しません**（`diffs/` には `from-phase-0` / `from-phase-1` / `from-chapter-1` のみ）。

### 記事との矛盾

| 記事の記述 | 実際 |
| --- | --- |
| 2-2「第3章の toggle/delete はまだ無くてよい。第2章 tag の内容に合わせる」 | 第2章 tag にはすでに toggle/delete/Realtime/sort が**入っている** |
| 2-2「`03-chapter2-cognito-auth/snapshots/index.ts` を `aws-blocks/index.ts` にコピー」 | コピーすると**第3章の中身が丸ごと入る** |
| 発展編「`04-chapter3-advanced/snapshots/index.ts` 等を参照して追記」 | 追記すべき差分が無い（第2章と同一） |
| 発展編「`git diff chapter-3-advanced`」で完成形と比較 | 第2章から始めると diff は**常に空** |

### 読者に起きること

- プロローグの説明（「第2章は Cognito だけ、toggle/delete は第3章」）を信じて手で書くと、`git diff chapter-2-cognito-auth` が**大量の差分**になり「自分が間違えた」と誤解する。
- 逆にスナップショットをコピーすると、第2章の時点で完成形になり、第3章でやることが消える。
- どちらにせよ「誰がやっても同じ結果」になりません。

### 修正案（いずれか）

1. **タグを切り直す（推奨）**: `chapter-2-cognito-auth` を「Cognito + userId 分離のみ（toggle/delete/Realtime/sort 無し）」に作り直し、第3章の機能は `chapter-3-advanced` だけに入れる。`from-chapter-2.patch` も生成する。
2. **記事を実態に合わせる**: 「第2章で完成形まで一気に作り、第3章は解説のみ」と章立てを変更し、2-2 の「toggle/delete はまだ無くてよい」を削除。発展編も「コードは第2章で完成済み。ここでは Realtime/楽観ロック/Index の仕組みを解説する」に書き換える。

> 案1の方が「章ごとに差分が増える」ハンズオンの建付けと整合します。

## 🟠 重要: `ensure-chapter2-users.sh` が推奨ログイン経路（`aws login`）で必ず失敗する

記事 2-5 は「**推奨（再現性が高い）**」として次を案内しています。

```bash
bash scripts/ensure-chapter2-users.sh
npm run verify:chapter2
```

### 事実（検証済み）

`scripts/ensure-chapter2-users.sh` は冒頭で `set -euo pipefail`（`-u` = 未定義変数でエラー）を設定したうえで、`--profile "$AWS_PROFILE"` を**無条件・素の変数参照**で使っています。

```bash
set -euo pipefail
...
aws cognito-idp admin-get-user --profile "$AWS_PROFILE" ...   # 12行目ほか計3箇所
```

`AWS_PROFILE` が未設定だと、bash は即座に停止します（実測）:

```
$ bash -c 'set -euo pipefail; echo "$AWS_PROFILE"'
bash: line 1: AWS_PROFILE: unbound variable   （exit 1）
```

- 記事の推奨経路は `aws login`（デフォルトプロファイル、`AWS_PROFILE` 不要）です。
- 記事自身も「名前付きプロファイルを使う場合のみ `.env.local` に `AWS_PROFILE`」と書いており、`aws login` だけの読者は `AWS_PROFILE` を設定しません。
- 新規 clone では `.env.local` も無い → `source` もされない → **`AWS_PROFILE` 未定義** → スクリプトが最初の行で落ちる。

参考: 同じリポジトリの `scripts/run-sandbox.sh` は正しく書けています（`${AWS_PROFILE:-}` でガードし、未設定なら `--profile` を付けない）。`ensure-chapter2-users.sh` だけがこのガードを欠いています。

### 修正案

`run-sandbox.sh` と同じパターンに揃える:

```bash
PROFILE_ARGS=()
if [[ -n "${AWS_PROFILE:-}" ]]; then
  PROFILE_ARGS=(--profile "$AWS_PROFILE")
fi
# 各 aws コマンドで: aws cognito-idp admin-get-user "${PROFILE_ARGS[@]}" --user-pool-id ...
```

> `npm run verify:chapter2`（`verify-chapter2-auth.ts`）はユーザーを**作成せず** `signIn` するだけなので、`ensure-chapter2-users.sh` の成功が前提です。ここが落ちると第2章の検証手順全体が再現できません。

## 🟡 中: `git checkout <tag>` で detached HEAD になる説明が無い

各章は `git checkout phase-1-blocks-scaffold` 等で**タグに直接 checkout** させますが、これは detached HEAD 状態です。

- 初心者は「今どのブランチにいるのか」「編集してよいのか」が分からない。
- そのまま編集して `git commit` すると、どのブランチからも参照されないコミットになり、別タグへ checkout した瞬間に**見失う**（`git switch -` でも戻れず、reflog を知らないと事実上ロスト）。
- 「誰がやっても同じ」を担保するうえで、作業ブランチの作り方は必須の前提知識です。

### 修正案

「ハンズオンの進め方」に一文追加:

```bash
git switch -c work/chapter1 phase-1-blocks-scaffold   # タグから作業ブランチを作る
```

または各章冒頭の `git checkout <tag>` を `git switch -c <章名> <tag>` に統一し、「detached HEAD で直接編集しない」と注記する。

## 🟡 中: `cognito-verifier.ts` の存在・役割が説明されない

第2章でコピーする `index.ts` は `import { CognitoVerifier } from './cognito-verifier.js';` を含みます。

- 事実（検証済み）: `cognito-verifier.ts` は `chapter-1-minimal-crud` タグの時点で**すでに存在**するため、import 自体は解決します（壊れません）。
- ただし記事には**この一文も無い**ため、初心者は「`./cognito-verifier.js` なんてファイル作った覚えがない、欠けているのでは？」と不安になります。`*.js` 拡張子で `*.ts` を import している点（NodeNext/ESM の作法）も初心者には引っかかりポイント。

### 修正案

2-2 か 2-3 に一言:「`cognito-verifier.ts` はリポジトリに同梱済み（JWT 検証ヘルパー）。新規作成は不要。`.js` 拡張子のまま import するのは ESM の解決ルールによるもの」。

## 🟡 中: 「編集中に Sandbox(watch) が自動再デプロイする」関係が明示されていない

「章ごとの dev モード」表は良くできていますが、**読者の編集がいつ AWS に反映されるか**の因果が言語化されていません。

- Phase 0 で `npm run sandbox`（watch）を起動したまま、第1章で `git checkout phase-1-blocks-scaffold` → `index.ts` を書き換える、という流れ。
- このとき watch 中の Sandbox は**ファイル変更を検知して Blocks Lambda を再デプロイ**します。つまり「ブラウザのデータが Sandbox DynamoDB に入る」のは、`client.js` が Sandbox API を向いていることに加え、**自分が編集した `index.ts` が Lambda として再デプロイされている**からです。
- さらに Phase 0 の baseline（Amplify のみ・Blocks 無し）で起動した watch のまま、Phase 1/第1章のコードへ checkout すると、watch が差分を検知して**作り直し**ます。タイミングによっては再デプロイ中の一時的なエラーやラグが出ます。

初心者は「保存先が DynamoDB になる」理由と「checkout したら Sandbox が勝手に作り直す」挙動を理解していないと、再現結果が揺れて見えます。

### 修正案

- 「章ごとの dev モード」直後に 1〜2 文:「`npm run sandbox` は watch。`index.ts` を編集・checkout するたびに Lambda が自動再デプロイされる。`[Sandbox] Watching... → ✔ Deployment completed` を待ってからブラウザを再読込する」。
- 章を切り替える際の安全な順序（例: 一度 Sandbox を止めてから checkout → 再起動、または再デプロイ完了を待つ）を明記。

## ⚪︎ 軽微

### a. Node バージョン表記の不一致
- 記事チェックリスト: 「Node.js **20.20+**」「Nix 利用時は v22 固定」
- リポジトリ `README.md`: 「無い場合は Node.js **22+**」
- `package.json` `engines`: `>= 20.20.0`

→ 致命的ではないが、非 Nix 読者向けに「20.20+ か 22+ か」を 1 つに統一すると親切。

### b. Nix の前提（flakes 有効化）が未案内
`nix develop` は flakes / `nix-command` の experimental features を要求します。未設定環境では
`error: experimental Nix feature 'nix-command' is disabled` が出ます。
→ 「Nix を初めて入れる読者向け」に、flakes 有効化（`~/.config/nix/nix.conf` に `experimental-features = nix-command flakes`）か、Determinate Installer 等の案内を一行。

### c. Playwright のブラウザバイナリ未案内
`npm run capture:screenshots` / 一部 verify は Playwright を使います。`npx playwright install` 相当でブラウザを入れないと初回失敗します。
→ スクショ取得を読者にやらせるなら、`npx playwright install chromium` を前提に追記。やらせないなら「スクショは著者取得済み・再実行不要」と明記。

### d. `aws login` の可用性
`aws login`（CLI 2.32+）は新しめのコマンドで、未対応バージョンや組織が一定数あります。代替（`aws sts get-caller-identity` が通れば可）は書かれていて良い対応です。本文の主動線を `aws login` 一択に見せず、「要は `get-caller-identity` が通ればよい」を先に置くと、つまずきが減ります。

### e. `git diff <tag>` での「答え合わせ」案内
🔴 #1 が未修整のままだと、第2章・第3章の `git diff` 比較がすべて誤解を生みます。#1 を直せば自然に解消します。
