# ARTICLE-DRAFT.md レビュー（初心者向けハンズオン / 再現性重視）

## 総評

現在の記事は、注意点の量は十分多いです。むしろ問題は「説明不足」だけではなく、「初心者がどの説明を今読むべきか判断しづらい」ことです。

AWS Blocks を知らず、Amplify を少し触った程度の読者に対しては、次の3点が特に詰まりやすいです。

1. 「どのブランチ・タグ・ディレクトリを触っているのか」が途中で曖昧になりやすい。
2. 「ローカルで動いているもの」と「AWS上で動いているもの」の境界が難しい。
3. 「コピーして動かす」手順はあるが、「なぜその変更で動くのか」の理解がまだ薄い。

ハンズオンとして公開するなら、最優先で直すべきなのは「成功ルートを一本道にすること」です。補足や例外は必要ですが、初心者向け本文では本流から外し、折りたたみ・補足・発展扱いにした方がよいです。

## 優先度 P0: 公開前に直した方がよい箇所

### 1. git clone が SSH 前提になっており、初心者が最初で止まりやすい

現在は次のコマンドになっています。

```bash
git clone git@github.com:k-adachi-01/hands-on-amplify-todo-to-aws-blocks.git
```

初心者向けなら SSH 鍵未設定で止まる可能性が高いです。AWS Blocks 以前に GitHub SSH の問題になり、ハンズオンの再現性を落とします。

推奨修正:

```bash
git clone https://github.com/k-adachi-01/hands-on-amplify-todo-to-aws-blocks.git
cd hands-on-amplify-todo-to-aws-blocks
```

SSH を使いたい人向けに補足で次を置く程度で十分です。

```bash
git clone git@github.com:k-adachi-01/hands-on-amplify-todo-to-aws-blocks.git
```

理由:

- 読者の前提は「Amplify を少し触ったことがある程度」であり、GitHub SSH 設定済みとは限らない。
- ハンズオンの最初の失敗は離脱率が高い。
- HTTPS clone なら誰でも再現しやすい。

### 2. Nix 推奨なのに本文の手順は Nix 必須に見える

記事冒頭では「Nix dev shell 推奨」と書いていますが、手順では毎回 `nix develop` を実行しています。そのため、Nix を使わない読者が「自分は対象外なのか」「Node 20.20+ を入れればよいのか」を判断しづらいです。

推奨修正:

冒頭でどちらかに寄せるべきです。

選択肢 A: 再現性を最優先し、Nix 必須にする。

```markdown
本記事では再現性を優先するため、Nix dev shell を前提に進めます。Nix を使わない場合は Node.js 20.20+ / npm 10.8+ を自分で用意できますが、以降の手順は Nix 利用時のものとして説明します。

選択肢 B: Nix あり / なしの手順を明確に分ける。

```markdown
Nix を使う場合:

node -v
npm -v
```

Nix を使わない場合:

```bash
node -v   # 20.20 以上
npm -v    # 10.8 以上
```
```

ハンズオンとしては A の方が再現性は高いです。ただし、初心者向け記事としては Nix 自体が別の学習コストになるため、「Nix が分からない人は何をすればよいか」だけは短く書くべきです。

### 5. 第1章の `src/App.tsx` の変更が「参考にする」だけなので、初心者には再現性が弱い

`aws-blocks/index.ts` は本文に完全なコードがあります。一方で `src/App.tsx` は「snapshot を参考に、少なくとも次を満たす」となっています。

これは初心者向けハンズオンとしては弱いです。読者によって実装差が出ます。

推奨修正:

第1章では `App.tsx` も全文掲載するか、少なくとも「変更前 / 変更後」の差分を載せてください。

最低限、次の粒度が必要です。

```markdown
Amplify 版では `client.models.Todo.observeQuery()` が Todo 一覧を購読していました。
第1章では Realtime はまだ扱わないため、`load()` で `api.listTodos()` を呼び、Todo 作成後に再取得します。
```

そのうえで、該当コードを載せます。

```typescript
async function load() {
  const items = await api.listTodos();
  setTodos(items);
}

async function createTodo() {
  await api.createTodo(content);
  setContent('');
  await load();
}
```

理由:

- 「参考にする」は読者の判断が入る。
- ハンズオンでは、誰がやっても同じコードになることが重要。
- Blocks の理解として一番重要なのは `observeQuery` から `api.listTodos()` への置き換えなので、ここは省略しない方がよい。

### 6. `git diff chapter-1-minimal-crud` は確認意図が曖昧

現在は各章の確認として次が出ています。

```bash

### 10. Amplify と Blocks の比較表に「今回どちらを使い続けるか」を追加する

現在の比較表はよいですが、この記事は Amplify を完全に捨てる内容ではありません。そのため、読者が「Amplify はどこに残るのか」を明示した方がよいです。

| 役割 | 今回使うもの | 説明 |
| --- | --- | --- |
| UI | React + Vite | ローカルで動かす画面 |
| ログイン画面 | Amplify UI Authenticator | Cognito のサインイン UI として使う |

### 11. `DistributedTable` のキー設計をもう少し説明する

第1章では `partitionKey: 'todoId'`、第2章では `userId: user.sub` を partition key にする説明があります。ただ、なぜこれでユーザー分離になるのかが初心者には見えにくいです。

推奨追加:

```markdown
第1章では、Todo ごとに一意な `todoId` を partition key にします。この時点ではユーザー情報を保存しないため、全員共通の Todo 一覧になります。

第2章では、ログイン済みユーザーの `sub` を `userId` として保存し、`userId` ごとに Todo を取得します。これにより、user-a で作った Todo は user-b の一覧に出なくなります。
```

可能なら、第2章のコード近くに次のようなミニ表を追加すると理解が揃います。

| 章 | partition key | 意味 |
| --- | --- | --- |
| 第1章 | `todoId` | Todo ごとに保存するだけ。ユーザー分離なし |
| 第2章 | `userId` | ログインユーザーごとに Todo を分ける |

### 12. `CognitoVerifier` の役割を「Authenticator との違い」として説明する

現在は `CognitoVerifier` + `auth.requireAuth(context)` と書かれていますが、初心者は「Authenticator でログインしているなら API 側の検証は不要では？」と考えやすいです。

推奨追加:

```markdown
`Authenticator` はブラウザ側のログイン UI です。画面上でログイン済みに見えても、API 側はそのリクエストが本当に Cognito で認証されたユーザーから来たものかを検証する必要があります。

そこで第2章では、Blocks API の中で `CognitoVerifier` を使い、リクエストに含まれる ID token を検証します。検証できた場合だけ `user.sub` を取り出し、そのユーザーの Todo として保存します。
```

理由:

- フロントエンド寄りの読者ほど、UI 認証と API 認可の境界で混乱しやすい。
- この記事の価値である「バックエンドの中身が見える」に直結する。

### 13. `aws-blocks/client.js` の正体をもう少し説明する

本文では `aws-blocks/client.js` が自動生成で編集しないと書かれています。これは重要ですが、初心者には「どこから import されるのか」「なぜ `import { api } from 'aws-blocks'` で使えるのか」が見えません。

推奨追加:

```markdown
`import { api } from 'aws-blocks'` で使う `api` は、`aws-blocks/index.ts` に書いた `ApiNamespace` から生成されるクライアントです。

ブラウザ側では `api.createTodo(...)` のように普通の関数呼び出しに見えますが、実際には `aws-blocks/client.js` が Blocks RPC のエンドポイントへ HTTP リクエストを送っています。
```

第2章ではさらに次を足すとよいです。

```markdown
第2章以降は、このリクエストに Cognito の ID token が付与されます。API 側では `CognitoVerifier` がその token を検証します。
```

## 最終判断

現状のままでも、ある程度 Git / AWS / Node に慣れた読者なら完走できる可能性は高いです。

ただし、想定読者を「AWS Blocks は知らない、Amplify は少し触ったことがある程度」とするなら、今のままだと次の層が詰まります。

- Git tag / branch に慣れていない人
- Nix を使ったことがない人
- AWS CLI 認証方式の違いが分からない人
- Authenticator と API 側認証の違いが分からない人
- ローカル dev server と Sandbox API の境界が分からない人

公開前に最低限直すなら、次の5つです。

1. clone は HTTPS をデフォルトにする。
2. Nix 必須なのか推奨なのかを統一する。
3. 本記事の推奨ルートだけを先に説明し、オフライン実行は補足へ回す。
4. 第1章の `App.tsx` 変更を全文または明確な差分で載せる。
5. 片付けを「任意」ではなく、初心者向けの必須手順に近い扱いにする。

この5つを直すだけで、初心者向けハンズオンとしての再現性はかなり上がります。
