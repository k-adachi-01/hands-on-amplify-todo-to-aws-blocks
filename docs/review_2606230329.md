## 総評

「概念マップ」「対応表」「トラブルシュート」はよく整理されていて、Amplify 経験者が Blocks の世界観を掴むのに有効です。一方で **ハンズオン（手を動かして同じ結果に到達する）としては、肝心の「何をどう書き換えるか」の手順が抜けています**。第1章・第2章は設計の要約とコード断片だけで、読者がゼロから同じコードに到達する道筋（全文 or `git checkout` の明示）がありません。

最大の問題は次の3点です。まずここを直さないと「再現性」は担保できません。

1. **第1章・第2章に "実際に編集する手順" がない**（コード全文も checkout 指示もない） → 🔴
2. **`npm run dev` が「ローカル mock」なのか「Sandbox 上の Lambda」なのか、資料間で矛盾**している → 🔴
3. **プレビュー版パッケージ（`@aws-blocks/*`）と `aws login` / `create-blocks-app` が、一般読者の環境で本当に入手・実行できるのか**が前提として明示されていない → 🔴

以下、詳細。

## 🔴 致命的（再現性を壊す）

### 1. 第1章・第2章に「手を動かす手順」が存在しない

記事の第1章は `aws-blocks/index.ts` を「定義し」、フロントを書き換える、と書いてありますが、

- 実際に書く `index.ts` の **全文がない**（断片のみ）
- `src/App.tsx` の差分が **まったくない**（`await api.createTodo(title)` の1行だけ）
- 「どのファイルのどこを、Amplify 版から何に置き換えるか」の **diff / before-after がない**

各章 README には `snapshots/aws-blocks-index.ts` や `snapshots/App.tsx` への参照がありますが、**記事本文からは辿れません**。これでは「読みもの」にはなっても「ハンズオン」になりません。

さらに深刻なのは **git tag の扱い**です。記事は各章末に `tag: chapter-1-minimal-crud` と添えるだけですが、リポジトリ HEAD の `aws-blocks/index.ts` は既に **第2/3章相当（`requireAuth` + Realtime + index 入り）の完成形**です（実ファイルを確認）。

```7:13:hands-on-walkthrough/aws-blocks/index.ts
const scope = new Scope('hands-on-todo');

const auth = new CognitoVerifier({
    userPoolId: process.env.COGNITO_USER_POOL_ID!,
    clientId: process.env.COGNITO_CLIENT_ID!,
    tokenUse: 'id',
});
```

つまり記事の第1章で「**まだログインは不要**」と言いながら、読者が clone 直後の HEAD で `npm run dev` すると **全 API が `requireAuth` 必須の第2章コード**で動きます。第1章の「ログイン不要で Todo 追加」は **HEAD では再現しません**。

**対応案（どちらか必須）:**
- (A) 各章の冒頭に明示的に `git checkout phase-1-blocks-scaffold` → 編集 → 実行、という **tag ベースの進め方**を据える。「対応する tag に戻ってから読む」ことを Phase 0 直後に1回宣言する。
- (B) 章ごとに **編集する全ファイルを記事本文に全文掲載**（最低でも `aws-blocks/index.ts` と `src/App.tsx`）。

個人的には (A) を主、(B) のうち `index.ts` の章別全文だけ補う、が現実的です。

### 2. `npm run dev` の実体が資料間で矛盾している（ハイブリッド構成の核心）

記事自身が「初心者が一番混乱しやすい点」と書いている箇所ですが、**資料間で説明が食い違っています**。

- 記事の Mermaid / 表: Blocks RPC は「**Sandbox 上の Lambda**」を叩く。だから「データは AWS 上の DynamoDB に保存され、消えない」（本文 200 行目）
- Phase 0 README（`00-.../README.md`）: 「Blocks API（開発中）= **ローカル dev server の mock**」「Blocks Lambda（本番相当）= `custom.blocks_api_url`」と**別物**として並記
- 第1章 README（`02-.../README.md`）:「ローカルは **`.bb-data/` に永続化（AWS 不要）**」

読者の素朴な疑問「**`npm run dev` したとき、createTodo はローカルの `.bb-data/` に書くのか、Sandbox の DynamoDB に書くのか**」に、記事は一意な答えを返せていません。しかもこれが「Todo が消えない理由」の説明の根拠になっているので、ここが曖昧だと **理解の再現性（同じ納得に至る）が崩れます**。

**対応案:**
- 「第1章 = AWS 不要・ローカル mock（`.bb-data/`）」「第2章 = Sandbox の Cognito + Sandbox API/DynamoDB に接続」と、**章ごとにモードを言い切る**。
- 記事冒頭の「Phase 0 から AWS アカウント必要」という設計と、第1章 README の「AWS 不要」が衝突している点も整理する（記事は Phase 0 で Sandbox を立てっぱなしにする方針なので、「第1章は本来 AWS 不要だが、本ハンズオンでは Phase 0 の Sandbox を立てたまま進める」と一言補えば矛盾が消える）。
- 「データが消えない理由」は、**そのモードで何がどこに保存されるか**を1文で確定させる。

### 3. プレビュー版パッケージ・新コマンドの「入手可能性」が前提として書かれていない

ハンズオンの土台すべてが、一般公開されていない可能性のあるものに依存しています。

- `package.json` の `@aws-blocks/blocks` / `@aws-blocks/bb-realtime` は **バージョン `"*"`**、`bb-distributed-table` も追加前提。**これらが公開 npm から `npm install` で入るのか**が読者の最初の関門。入らなければ「リポジトリの準備」の `npm install` 時点で全員が詰まります。
- `npx @aws-blocks/create-blocks-app@latest`（Phase 1）も同様に、**プレビューアクセス無しで実行できるか**不明。
- `aws login`（AWS CLI 2.32.0+）も新しめのコマンド。古い CLI の読者は使えません。

記事は「プレビューです」と性質には触れていますが、**「動かすのに何（アカウント種別・プレビュー登録・private registry・最低バージョン）が必要か」という再現の前提条件**を書いていません。

**対応案:** 「始める前のチェックリスト」直後に、
- プレビューパッケージの入手方法（公開 npm で入るのか／要申込みなのか）
- 入らなかった場合にどうなるか（=ここで止まる、という明示）
- `aws login` が使えない場合の代替（後述）
を**正直に**明記する。再現性ハンズオンでは「動かない前提条件」を隠さないことが最重要です。

## 🟡 要改善（つまずき・誤解の原因）

### 4. UI でのサインアップが `example.com` だとメール確認で詰まる

第2章「手順（UI で確認する場合）」では `user-a@example.com` を **Create Account** で登録する流れですが、Cognito の標準サインアップは **確認コードをメール送信**します。`example.com` は受信できないため、**UI 経路は確認コード入力で止まります**。

第2章 README には「メール確認コードが必要な場合は…」と注記があり、CLI 経路（`ensure-chapter2-users.sh` で auto-confirm）が用意されているのに、**記事本文の UI 手順にはこの注意が抜けています**。初心者は UI 手順を素直に試して確実に詰まります。

**対応案:**
- 記事の UI 手順に「`example.com` 宛は受信できない。**自分の実メール**を使うか、メール無しで進めたい場合は CLI 手順（`ensure-chapter2-users.sh`）を使う」と明記。
- 「同じ結果を確実に再現したいなら CLI 手順を推奨」と一言添える。

### 5. 環境変数 `COGNITO_USER_POOL_ID` / `COGNITO_CLIENT_ID` の供給元が未説明

`aws-blocks/index.ts` は `process.env.COGNITO_USER_POOL_ID` / `CLIENT_ID` を直接参照します（実ファイル 9–13 行）。しかし記事は **これらがいつ・どこで・どう設定されるか**を一切説明していません。読者が `npm run dev` した時、これらが空なら `requireAuth` 系が壊れます。

Sandbox / `generate-client` が注入している想定だと思いますが、**ブラックボックスのまま**だと「なぜ動くのか」が再現できません（理解の再現性が落ちる）。

**対応案:** 第2章で「これらの env は `npm run dev` / `npm run sandbox` が `amplify_outputs.json` から流し込む（手で設定しない）」のように、**供給経路を1文で確定**させる。手で設定が要るなら手順を書く。

### 6. Phase 0 の起動順序（A→B）の依存関係が暗黙

`npm run dev`（`server.ts`）は `amplify_outputs.json` から `client.js` を生成します。`amplify_outputs.json` は **Sandbox デプロイ完了後**にしか存在しません。

記事は「ターミナル A で sandbox → ターミナル B で dev」の順に並べてはいますが、**「A の `amplify_outputs.json` 生成（`✔ Deployment completed` / `File written: amplify_outputs.json`）を見届けてから B を起動」という依存を明示していません**。せっかちな読者は同時起動して B 側で失敗します。

**対応案:** 0-3 の頭に「**0-2 で `File written: amplify_outputs.json` を確認してから**新ターミナルへ」と1行。

### 7. `aws login` の前提（リージョン・初期設定・代替手段）

- `aws login` 自体が `aws --version 2.32.0+` 前提だが、**古い場合の更新方法**が無い（「2.32.0 以上であること」だけ）。最低限リンクか1行が欲しい。
- `aws login` は**デフォルトリージョンを設定しない**ことがある。Sandbox はリージョン必須なので、トラブルシュート任せにせず **`aws configure set region ap-northeast-1`（任意の対象リージョン）を事前手順**に入れるのが安全。
- そもそも `aws login` が未設定/未対応の組織アカウントの読者向けに「アクセスキー or SSO など従来手段でも `aws sts get-caller-identity` が通ればよい」という**逃げ道**を1行補うと再現性が上がる。

### 8. 「第1章」の `listTodos` と実コードの乖離

記事第1章は `setTodos(await api.listTodos())` ですが、実 `index.ts` の `listTodos(sortBy?)` は内部で `requireAuth` 必須・`query`（`collectAsync`）です。第1章 tag では違う実装のはず。**どの段階のコードを見せているのか**が読者に伝わらないと「自分の手元と違う」混乱を生みます（→ 課題1と同根）。

### 9. Node バージョン表記の揺れ

- 記事チェックリスト: 「Node.js 22+」「npm 10+」
- `package.json` engines: `node >= 20.20.0` / `npm >= 10.8.0`
- 実行記録: v22.22.3 / npm 10.9.8

致命的ではないが、**「22 必須」なのか「20 でも動く」なのか**を統一。再現環境を固定したいなら「Nix の dev shell で固定された v22 を使う（手動環境なら 20.20+ でも可）」と明記。

### 10. Phase 1 の `create-blocks-app .` を読者が実行すべきか曖昧

記事は「本リポジトリでは済み」としつつコマンドを提示。読者が良かれと思って既存リポジトリ上で再実行すると、scaffold が二重生成 or 既存ファイルを上書きする恐れ。**「本ハンズオンでは実行しない（参考情報）」**とはっきり書く。

## 🟢 軽微（表現・体裁）

- **L113「自分の AWS 身份」**: 「身份」は中国語。日本語なら「自分の AWS の身元（ID）」「呼び出し元」等に。
- **L138「片付け: `npm run sandbox:delete`」/ L448 構成案では `sandbox:destroy`**: `package.json` のスクリプトは `sandbox:delete`。記事は正しいが、構成案・本文・まとめで `delete`/`destroy` が混在しないか全体を grep して統一。
- **アンカーリンク `#aws-へのログインaws-login`（L224）**: 日本語見出し＋全角括弧＋バッククォートからの自動生成アンカー。GitHub 表示で実際に飛ぶか1度確認を。
- **所要時間**: 記事「Sandbox 約4〜5分」 vs README「約271秒」。整合（≒4.5分）しているので問題なし。ただし `nix develop` 初回ビルドと `npm install` の所要も「数分」と幅を持たせて書くと親切。
- **`verify:chapter2` の説明（L402 トラブルシュート）**: 「Amplify の SRP `signIn` を使う」は正確（README 65 行と一致）。良い。
- **第3章 Realtime「Sandbox では環境により未配線」**: 発展編として割り切るのは妥当。ただし「未配線だと subscribeTodos が何を返す/投げるのか」を1行補うと、読者が `load()` フォールバックに自然に流れられる。
